# EC AI プロダクト構想（事業コンセプト）

> このドキュメントは「何を作るか」ではなく **「何のために作るか」** の合意事項。
> 実装や優先順位の判断に迷ったら、まずここに立ち返る。
> 技術的な作り方・アーキテクチャは `CLAUDE.md` を参照。

## 事業概要

私たちは楽天市場向けのAIを開発している。
ただし、本質的には「楽天分析ツール」を作りたいわけではない。

目指しているのは、**ECコンサルティングをAIで民主化すること**。

## 課題

EC業界では、売上の差は商品の差ではなく **「意思決定の質」** によって生まれることが多い。

大手企業は優秀なECコンサルタントを雇い、以下を日々改善している。

- SEO
- 広告
- レビュー
- 商品ページ
- 価格
- イベント

一方で中小企業は「何を改善すればいいか分からない」という状態で運営しているケースが非常に多い。
結果として、経験や勘に依存した店舗運営になっている。

## 解決したいこと

高額なECコンサルを受けられる会社だけが成長する世界ではなく、
**AIが誰にでも優秀なECコンサルの役割を提供する世界** を作りたい。

つまり、EC事業者が毎日「今日やるべきこと」をAIから受け取り、
改善を積み重ねることで売上を伸ばせる世界。

## プロダクトコンセプト — AIストアマネージャー

これは分析ツールではない。目指しているのは **AIストアマネージャー**。

ダッシュボードを見せることが目的ではない。
店舗が **「次に何をすればいいか」を判断できる状態** を作る。

## ロードマップ

### MVP（Phase 1）— 今日やるべきことを提案する

毎日AIが「今日やるべきこと」を提案する。例:

- この商品画像を変更してください
- この商品名を改善してください
- レビュー返信してください
- 広告予算を変更してください
- 競合価格が変わりました

重要なのは **分析ではなくアクション**。

### Phase 2 — 学習ループ

```
提案 → 実施結果 → 売上変化 → 学習
```

これを繰り返し、改善提案がどんどん賢くなる。

### Phase 3 — 半自動・自動実行

提案だけではなく、楽天API等と連携して以下を半自動・自動で実行する。

- 商品名変更
- SEO変更
- 広告調整
- クーポン発行

最終的には **EC店舗のAI運営責任者** になる。

## 最初のターゲット

**楽天市場 × 家具業界**

理由: 家具業界で強いネットワークがあり、最初の成功事例を作りやすいため。

導入候補:

- アイズコーポレーション
- DHI
- LOWYA
- 松田家具
- 関家具

まず家具カテゴリで圧倒的なシェアを作る。その後、横展開する。

```
家具 → インテリア → 生活雑貨 → 家電 → 楽天全体 → Amazon → Shopify
```

## 営業戦略

最初は既存ネットワークで導入する。
**アイズコーポレーションは「顧客」ではなく「共同開発パートナー」** として改善を繰り返す。

成功事例を作った後、代理店販売へ移行する。代理店候補:

- EC制作会社
- 楽天コンサル会社
- 広告代理店
- 制作会社

紹介と代理店を成長エンジンにする。

## KPI

追うべき数字は **導入社数ではない**。まずは **成功事例5社**。

定量化する指標の例:

- 売上改善
- CVR改善
- 広告費削減
- 利益率改善

この成功事例がそのまま営業資料になる。

## 将来像

## この構想が実装に与える示唆（派生メモ）

上記の構想から導かれる、コードを書くときの判断基準。構想本体と違い、
これは実装が進むにつれて更新されるべきもの。

**1. 出力の最終地点は「数値」ではなく「次のアクション」**

画面やAPIを追加するとき、「この数字を見た人は次に何をすればいいか分かるか」を必ず問う。
既存の改善重要アラート・KPI評価マトリクス・アクセス逆算パネルは、この方向に沿った資産。
数値を並べただけで終わる画面は、構想上は未完成とみなす。

**2. データが無いときこそ、意思決定を止めない**

「データがありません」で画面全体を隠すのは、この構想では最も避けるべき挙動。
実際に2026-07で、店舗全体の売上145万円という判断材料があるのに、RPP広告データが
未取込というだけで全画面が空になる不具合が起きた（修正済み）。

原則: **「今わかること」と「まだわからないこと」を切り分けて提示する。**
優秀なコンサルがやるのはまさにこれで、欠損を理由に沈黙しない。

**3. モール横展開に備え、取込み層とロジック層を分ける**

将来 Amazon / Yahoo / Shopify へ広げる前提なので、
「モール固有のデータ取込み・正規化」と「KPI計算・判定ロジック」を混ぜない。
`backend/calculations.py` が全KPI計算の単一の真実になっている現状は、この方針に合致している。
モール固有の事情（RPP、商品分析レポート等）は取込み層に閉じ込める。

**4. Phase 2 の学習ループにはデータ構造が要る**

「提案 → 実施 → 売上変化 → 学習」を回すには、提案とその実施結果・その後の数値変化を
紐づけて永続化する必要がある。`RppActionCheck`（management_no ベースのチェック保存）は
その萌芽。Phase 2 に入る際はここを正式なデータモデルへ育てる。

**5. 成功事例5社がKPI＝効果の定量化はプロダクト機能**

売上改善・CVR改善・広告費削減・利益率改善を「導入前後で」定量化して出力できると、
そのまま営業資料になる。レポート出力機能はこの観点で設計する価値がある。
