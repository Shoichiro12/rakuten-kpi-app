## 1. 今日やったことの全体像

「マスタテーブル追加」の実装指示書からスタートし、それを土台に在庫・原価・廃盤まわりの機能を積み上げた。

1. マスタ4テーブル追加（shops / product_categories / products / product_costs）
2. 原価マスタと原価の焼き込み（cost_of_sales を実値化）
3. is_active（廃盤）を分析全体に反映
4. 在庫の自動連携（No.4）
5. 商品マスタ管理画面（新規ページ /master）
6. カテゴリ管理（作成・リネーム・削除）
7. 商品マスタのCSV一括入出力
8. 先読み発注アラート（欠品前）
9. 在庫アラート一覧
10. 発注アラート閾値の店舗設定化

## 2. 設計上の前提（必読）

- **このリポジトリはマルチテナント。** 全モデルが `UserScopedMixin`（`user_id`列）を継承し、
  `tenancy.py` が全クエリをログインユーザー単位に自動で絞り込む。新規テーブル4つも同様に対応済み。
  → 新しいモデルを足すときは必ず `UserScopedMixin` を継承すること。
- **元の指示書は単一店舗SQLite前提だったが、実コードに合わせてマルチテナント化して実装した。**
  - 4テーブルとも `user_id` 付き・ユニーク制約は `user_id` 込み
  - デフォルト店舗は「起動時に全体1行」ではなく、本番は各ユーザー初回アクセス時に遅延生成
    （`masters.get_or_create_default_shop`）。ローカル（認証無効）のみ起動時に1行投入。
  - `Product.shop_id` は固定1ではなく現ユーザーのデフォルト店舗を解決して入れる
- **No.1 評価マトリクス / No.2 YoY統一ロジックは既に実装済みだった**（`evaluation.py`）。
  今日はそこは触らず、`products`/`is_active` を土台にする未実装分だけをやった。
- マスタ管理APIは既存 `/api/products`（商品別KPI集計）と衝突するため `/api/master/*` に分離。

## 4. 追加したAPI一覧

```
GET  /api/costs                         原価一覧（個別/既定の区分つき）
PUT  /api/costs/default                 店舗デフォルト原価率＋全再計算
PUT  /api/costs/{management_no}          商品別原価率＋対象再計算
POST /api/costs/recalc                  現在の率でRppWeekly再計算
POST /api/costs/import                  原価CSV一括登録（管理番号,原価率）

GET  /api/master/products               商品マスタ一覧（is_active/category_idで絞込）
PUT  /api/master/products/{mno}          商品名/カテゴリ/is_active 編集
GET  /api/master/products/export         商品マスタCSV（BOM付き）
POST /api/master/products/import         商品マスタCSV一括取込（upsert）
GET  /api/master/categories             カテゴリ一覧
POST /api/master/categories             カテゴリ作成（find-or-create）
PUT  /api/master/categories/{id}         カテゴリのリネーム
DELETE /api/master/categories/{id}       カテゴリ削除（参照商品は未分類化）

GET  /api/shops/me                      店舗設定取得（原価率/経費率/発注閾値）
PUT  /api/shops/me                      店舗設定更新（原価率変更は再計算）

GET  /api/inventory/alerts              欠品・在庫僅少を機会損失順（廃盤除外）
```

## 5. 明日デプロイ/起動時の注意

- 既存のローカルSQLite（`backend/rakuten_kpi.db`）を使い続ける場合、起動時に
  `create_all`（新テーブル生成）＋`run_migrations`（`restock_lead_days` 列追加）が自動で走るので手動作業は不要。
- **過去データから商品マスタを埋めるには backfill を1回実行する**（自動同期はこれ以降の新規取込にしか効かない）:
  ```
  cd backend
  py -3 scripts/backfill_products.py            # ローカル/単一テナント
  # 本番の特定ユーザー分は: py -3 scripts/backfill_products.py --user-id <UUID>
  ```
- 本番（Supabase/Postgres）では `migrations._enforce_rls_pg` が全publicテーブルにRLSを自動適用するため、
  新4テーブルも保護される。デプロイ後に `GET /api/security-status` の `unprotected` が空であることを確認。

## 8. 開発メモ（既存の落とし穴、再掲）

- `vite.config.ts` のプロキシ先は `http://127.0.0.1:8000`（`localhost` に戻さない）。
- 「Failed to fetch」の典型原因は古い uvicorn がポート8000を掴んだまま。まず `:8000` を止めて再起動。
- クエリの列挙値は `typing.Literal[...]`（`Query(..., enum=[...])` はPydantic v2で効かない）。
- 全エンドポイントは常にJSONを返す（データ無しでも `{}`/`[]`）。フロントはこれに依存。
